# Test Plan

## Ticket

GitHub Issue #29: 収集対象 X アカウントの管理方法変更  
https://github.com/so-ota-biz/ai-agent-orchestration-settings/issues/29

## Test Targets

### 主要テスト対象
- 新しい設定ファイル読み込み機能 (`src/lib/config.ts`)
- YAML設定ファイルのパース処理
- 後方互換性（既存 `TARGET_ACCOUNTS` 環境変数との共存）
- 設定バリデーション機能
- エラーハンドリング

### 副次的テスト対象
- 既存のRSS取得処理への影響なし
- 型定義の整合性
- 設定読み込み後の `targetAccounts` 配列の形式

## Test Cases

### 1. 新設定ファイル読み込みテスト
- **TC-29-001**: 有効な `config/accounts.yml` が存在する場合、正常に読み込まれる
- **TC-29-002**: `config/accounts.yml` の各アカウント情報（name, description, category）が正しく取得される
- **TC-29-003**: 複数アカウントが設定された場合、全てが `targetAccounts` 配列に含まれる

### 2. YAML形式バリデーションテスト
- **TC-29-004**: 正しいYAML構文の設定ファイルが正常にパースされる
- **TC-29-005**: 不正なYAML構文（インデント誤り等）の場合、適切なエラーメッセージが表示される
- **TC-29-006**: 必須フィールド（name）が欠損している場合、バリデーションエラーが発生する

### 3. 後方互換性テスト
- **TC-29-007**: `TARGET_ACCOUNTS` 環境変数のみ設定時、従来通りの動作を維持
- **TC-29-008**: 両方設定時、環境変数が優先される
- **TC-29-009**: どちらも設定されていない場合、適切なエラーメッセージが表示される

### 4. エラーハンドリングテスト
- **TC-29-010**: 設定ファイルが存在しないが環境変数あり → 環境変数使用
- **TC-29-011**: 設定ファイル読み込みエラー時の適切なエラー出力
- **TC-29-012**: 空の設定ファイルの場合のハンドリング

### 5. 統合テスト
- **TC-29-013**: 新設定でのRSS取得処理が正常動作
- **TC-29-014**: `npm run run` コマンドでの全体動作確認
- **TC-29-015**: 設定変更後の既存テスト群への影響なし

## Risks / Unknowns

### テスト実行リスク
- YAMLライブラリ（js-yaml等）の追加によるテスト環境への影響
- 既存の `config.test.ts` の大幅変更が必要になる可能性

### 考慮事項
- 設定ファイルの存在確認とファイルシステムアクセスのテスト方法
- モックファイルシステムの使用検討
- CI/CD環境での設定ファイルテストの実行方法

### 未確定事項
- 設定ファイルの配置場所（プロジェクトルート/config/かsrc/config/か）
- デフォルト設定ファイルのバンドル方法
- 設定ファイルの更新監視機能の必要性（現在は不要と判断）
